# 实施方案：美国政治新闻聚合项目

## 目标
把当前“多脚本抓取 + JSON 归档 + 事后分析”的实现，收敛成一条稳定的新闻流水线：统一配置、安全获取、抓取入库、分析出报、定时运行。

## 现状判断
- 入口分散：`rss_monitor.py`、`newsapi_fetch.py`、`gdelt_fetch.py`、`fetch_tianapi.py` 都是独立脚本，彼此重复。
- 分层已出现但未收敛：`database.py` 和 `fetchers.py` 已有统一存储层与抓取层，`daily_run.sh` 仍在调用旧脚本。
- 分析仍依赖文件扫目录：`analyze_news.py` 直接读取 `data/` 下的 JSON，而不是数据库。
- 配置与密钥有风险：`daily_run.sh` 中硬编码了 `NEWSAPI_KEY`，`config.yaml` 里还有语法错误（Axios URL 未闭合）。

## 架构决策
- **SQLite 作为单一事实源**：抓取结果先入库，JSON 仅作为导出产物。
- **统一抓取入口**：所有数据源都走同一套 runner，不再由定时脚本直接调用多个散落脚本。
- **配置强校验**：启动时验证必需环境变量和 YAML 结构，失败则直接退出。
- **分析只读数据库**：报告生成只从数据库取数，不再扫描历史文件。
- **旧脚本保留兼容**：迁移期间可保留 wrapper，避免一次性切断现有流程。

## 依赖图
```
配置/密钥
   ↓
数据库（SQLite）
   ↓
统一抓取层（fetchers）
   ↓
统一运行入口
   ↓
分析与报告
   ↓
定时执行与导出
```

## 风险与缓解
| 风险 | 影响 | 缓解 |
|---|---|---|
| 外部数据源不稳定 | 抓取结果波动大 | 统一超时、重试、降级，保留源级日志 |
| 旧脚本与新入口并存 | 流程混乱、重复运行 | 先保留 wrapper，再逐步切换定时入口 |
| 配置不一致 | 启动失败或漏抓 | 启动时强校验，并给出明确错误 |
| 报告逻辑依赖历史 JSON | 迁移期间结果不一致 | 先保留导出，再逐步切换到数据库读源 |
| 缺少测试 | 回归难发现 | 先补核心纯函数和数据库测试 |
