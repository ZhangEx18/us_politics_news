两数之和：用哈希表将O(n²)暴力降为O(n)

入门
数组哈希表
如何在无序数组中，高效地找出和为目标值的两个数。

直觉理解

想象一下，你有一堆写着不同数字的卡片，你的任务是从中找出两张，让它们的和等于一个特定的数，比如10。最笨的方法是什么？拿着第一张卡片，然后一张一张地跟后面所有卡片去试，看哪两张加起来是10；接着再拿起第二张卡片，重复这个过程。这就像在电话簿里找某个人的号码，你从头到尾一页页翻，效率非常低。更聪明的做法是，当你看到一张卡片数字是 `3` 时，你立刻知道，你需要找的另一张卡片一定是 `7`（因为 10-3=7）。这时，你不需要再一张张翻电话簿了，你希望有一个魔法索引，能直接告诉你 `7` 在哪里，或者至少能立刻确认它存不存在。哈希表（在Python里就是字典）就是这个魔法索引。你每看一张卡片，就把它和它对应的位置（索引）记到这个索引本上。这样，当你需要找 `7` 时，你只需在索引本里一查，如果查到了，就说明找到了；如果没查到，说明这张卡片暂时还没出现过，你只需要继续处理下一张卡片。这个过程，本质上是用一个额外的空间（索引本）来换取查找时间的巨大缩短，这就是“用空间换时间”的经典思路。

算法思路

解题框架可以分为两步：暴力枚举和哈希表优化。 **步骤一：暴力枚举（理解基础问题）** 最直接的想法是双重循环。我们遍历数组中的每一个元素 `nums[i]`，对于每个 `nums[i]`，再遍历它后面的所有元素 `nums[j]`（其中 `j > i`），检查 `nums[i] + nums[j] == target` 是否成立。如果成立，就返回它们的下标 `[i, j]`。这个方法时间复杂度是 O(n²)，对于数据量大的数组来说太慢了。 **步骤二：哈希表优化（一次遍历）** 我们观察到，暴力解法中，对于每个 `nums[i]`，我们都在其后面的序列里寻找一个值为 `target - nums[i]` 的数。这个寻找过程是 O(n) 的，这正是瓶颈所在。哈希表可以帮我们把这个寻找过程降到 O(1)。 具体做法是：我们遍历数组，对于当前元素 `nums[i]`，先计算它的“伴侣”值 `complement = target - nums[i]`。然后在哈希表（字典）里查找这个 `complement` 是否存在。如果存在，说明我们之前已经遇到过这个伴侣值，直接返回它的下标和当前下标 `i`。如果不存在，就把当前元素 `nums[i]` 作为键，它的下标 `i` 作为值，存入哈希表，然后继续遍历下一个元素。这样，我们只需要一次遍历，就能解决问题。

Python 示例

def twoSum(nums, target):
    """
    使用哈希表在一次遍历内解决问题
    """
    # 创建一个哈希表（字典），用于存储已经遍历过的数字及其索引
    hash_map = {}
    # 遍历数组，同时获取索引和值
    for i, num in enumerate(nums):
        # 计算当前数字需要的“伴侣”数字
        complement = target - num
        # 检查这个伴侣数字是否已经在哈希表中
        if complement in hash_map:
            # 如果找到了，直接返回两个索引
            return [hash_map[complement], i]
        # 如果没找到，将当前数字及其索引存入哈希表
        hash_map[num] = i
    # 根据题目保证，总会有解，所以这里理论上不会执行到
    return []

# 测试用例
if __name__ == "__main__":
    # 测试案例1
    nums1 = [2, 7, 11, 15]
    target1 = 9
    print(f"输入: {nums1}, 目标: {target1}")
    print(f"输出: {twoSum(nums1, target1)}") # 期望: [0, 1]
    
    # 测试案例2
    nums2 = [3, 2, 4]
    target2 = 6
    print(f"\n输入: {nums2}, 目标: {target2}")
    print(f"输出: {twoSum(nums2, target2)}") # 期望: [1, 2]
    
    # 测试案例3：注意下标不能重复使用同一个元素
    nums3 = [3, 3]
    target3 = 6
    print(f"\n输入: {nums3}, 目标: {target3}")
    print(f"输出: {twoSum(nums3, target3)}") # 期望: [0, 1]

C++ 对照

#include <vector>
#include <unordered_map>
using namespace std;

class Solution {
public:
    vector<int> twoSum(vector<int>& nums, int target) {
        // 创建一个哈希表，键是数字，值是它的索引
        unordered_map<int, int> hash_map;
        // 遍历数组
        for (int i = 0; i < nums.size(); i++) {
            // 计算伴侣值
            int complement = target - nums[i];
            // 在哈希表中查找伴侣值
            auto it = hash_map.find(complement);
            if (it != hash_map.end()) {
                // 如果找到，返回结果
                return {it->second, i};
            }
            // 如果没找到，将当前数字和索引存入哈希表
            hash_map[nums[i]] = i;
        }
        // 题目保证有解，这里返回空向量（作为占位）
        return {};
    }
};

复杂度分析

时间复杂度：O(n)。我们只对数组进行了一次遍历，n是数组的长度。在循环中，哈希表的查找和插入操作平均时间复杂度都是O(1)。 空间复杂度：O(n)。在最坏的情况下，我们需要将数组中的每一个元素都存入哈希表中，直到找到解或遍历结束，因此哈希表最多需要存储 n 个元素。这就是用空间换时间的典型代价。

易错点

  1. **陷阱1：同一元素重复使用。** 错误做法是在遍历时，先将当前元素放入哈希表，然后再去查找伴侣值。例如输入`[3,2,4]`,目标`6`，当`i=0`时，`num=3`，伴侣`3`，哈希表中刚刚放入了`(3,0)`，就会错误地返回`[0,0]`。**修复方法：** 必须遵循“先查找，后插入”的顺序。先检查伴侣值是否存在于之前元素（不包括当前元素）构成的哈希表中，找不到再将当前元素插入。
  2. 陷阱2：不处理哈希表的初始状态。** 对于输入`[3,3]`,目标`6`，第一个`3`遍历时，哈希表为空，找不到伴侣`3`，然后插入`(3,0)`。第二个`3`遍历时，就能在哈希表中找到第一个`3`及其索引`0`，正确返回`[0,1]`。如果代码逻辑错误，可能会误判。
  3. 陷阱3：未考虑存在重复元素的情况。** 虽然题目保证有且只有一个答案，但数组中可能存在多个相同的数字。我们的解法通过存储索引，天然地可以处理这种情况，因为每个索引都是唯一的。关键点在于我们存的是索引，而不是简单地标记数字是否存在。

延伸练习

  1. **练习1（巩固）：两数之和 II - 输入有序数组** 给你一个下标从1开始的整数数组 `numbers` ，该数组已按 **非递减顺序排列** ，请你从数组中找出满足相加之和等于目标数 `target` 的两个数。如果设这两个数分别是 `numbers[index1]` 和 `numbers[index2]` ，则 `1 <= index1 < index2 <= numbers.length`。请返回这两个数的下标。**思考：** 既然数组是有序的，除了哈希表，有没有更节省空间的双指针解法？
  2. 练习2（拓展）：三数之和** 给你一个整数数组 `nums` ，判断是否存在三元组 `[nums[i], nums[j], nums[k]]` 满足 `i != j`、`i != k` 且 `j != k` ，同时还满足 `nums[i] + nums[j] + nums[k] == 0`。请返回所有和为 0 且不重复的三元组。**思考：** 如何将“两数之和”的哈希表思路扩展到“三数之和”？（提示：固定一个数，问题就变成了在剩余数组中找两数之和）。注意去重技巧。
